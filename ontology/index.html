AIP Tool-use over the Ontology

A simulated agent. It holds a typed catalog of objects, links, and actions (the same schema an LLM would consume), matches your prompt against an intent table, and dispatches a real Action through the same middleware pipeline as the UI buttons. No model in the loop — just the wiring.

What this demonstrates

      
    
        
  • Facade / Registry — Ontology is the single entry point; object types are registered declaratively, link types via the LinkCoordinator.
    • 
        
  • LinkCoordinator — relationship logic is decoupled from objects: the coordinator owns a directed graph of declared links and resolves both forward FK joins and reverse property scans. Try links.outgoingFor('Flight') in the console.
    • 
        
  • Adapter — BackingAdapter defines a small interface (load(), getRow(), allRows()); JSONAdapter fetches static JSON, DuckDBAdapter registers the same JSON in DuckDB-Wasm's virtual FS and queries it via read_json_auto. Both populate Ontology.cache, so every read path (ObjectProxy, computed properties, integrity scan) is unaware of the source. Per-environment selection lives in env/{env}/manifest.json under adapters; dev uses JSON for fast iteration, staging and prod push Flight through DuckDB. Try ontology.objectTypes.get('Flight').adapter.
    • 
        
  • OLAP escape hatch — QueryBuilder.sql(text, params) runs raw parameterized SQL against DuckDB-Wasm; results route through Ontology.get(type, pk) so the kinetic-edit overlay, security read filter, and time-travel context all still apply. Try await query().from('Flight').sql("SELECT * FROM flights WHERE status = ?", ['Scheduled']).collect().
    • 
        
  • Proxy — ObjectProxy lazily merges base rows with kinetic edits on every property read.
    • 
        
  • Command — every mutation goes through ActionEngine.dispatch; validation, state machine, and side effects are encapsulated in the action spec.
    • 
        
  • Chain of Responsibility — dispatch runs a middleware pipeline (resolve → validate → preprocess → computeChanges → constraints → buildChangeSet → persist → effects). Plug your own with actions.use(handler); effect handlers run after persistence and see the committed ChangeSet. The toast notifications and console logs are wired this way.
    • 
        
  • Constraint Engine — declarative cross-object invariants registered on the Ontology and enforced by a pre-persist middleware. Demo constraints: originDestinationDistinct, pilotSingleScheduledAssignment, departureNotInPast. Try reassigning N101AA to P002 — it'll be rejected because P002 already holds the Scheduled flight N107AA.
    • 
        
  • Computed properties — derivation functions registered per (type, prop). ObjectProxy intercepts reads of computed names and evaluates inside a dependency-tracking scope; results memoize by recorded deps and cascade-invalidate on action/undo. Examples: Pilot.experience_tier, Flight.pilot_name, Pilot.assigned_flight_count. Try computed.stats().
    • 
        
  • Semantic branching — BranchManager tracks the active branch; ChangeSets carry a branchId. Reading from a branch overlays it on top of main; writing while on a branch isolates new actions there. Use the topbar to create a "what-if" branch, dispatch some actions, switch back to main and watch them disappear, then merge to promote them. Console: branches.list().
    • 
        
  • Recursive graph queries — fluent builder over LinkCoordinator: ontology.query().from(type).where(p).traverse(link).collect(). Cycle protection via per-step visited-sets and a max-depth limit. AIP intents like pilots to LAX and destinations of P001 compile to multi-hop traversals. Try query().from('Flight').where(f => f.status === 'Scheduled').traverse('flight_pilot').collect().
    • 
        
  • Memento — ChangeSets persist to localStorage; each is addressable and reversible.
    • 
        
  • Observer — UI subscribes to action/undo/loaded events; no direct DOM coupling to mutations.
    • 
        
  • State — Flight.status transitions are gated by the action validators (Scheduled → InAir → Landed, with Cancelled as an early exit).
    • 
        
  • Bi-temporal — every base row and ChangeSet carries created_at (transaction time) and valid_from (valid time). Reads accept an optional { asOfTx, asOfValid } context that filters out records the system wouldn't have known yet. The time-travel slider is the user-facing surface; the kernel does the same filtering for any caller.
    • 
        
  • AIP / tool-use — Ontology, LinkCoordinator, and ActionEngine each export a getSchema(); the Agent aggregates them into the catalog an LLM would see, then dispatches typed actions through the existing pipeline. Try agent.getSchema() in the console.
    • 
        
  • CRDT multi-tab sync — every ChangeSet carries a Lamport clock and the writer's nodeId (per-tab, sessionStorage). mergeEdits replays in (lamport, nodeId) order — deterministic across tabs even with concurrent writes. BroadcastChannel publishes locally-committed ChangeSets to peer tabs, which advance their clock to max(local, remote) + 1. Open this page in two tabs and dispatch from each — the topbar shows your nodeId · clk. storage events cover the case where BroadcastChannel is unavailable.
    • 
        
  • Data integrity worker — a Web Worker scans base data + ChangeSets out-of-band for orphan FKs, orphan ChangeSets, and kinetic drift. The kernel doesn't know about it; it subscribes to action/undo/loaded events and posts findings back to the main thread. Click Inject test issue to see drift detection in action.
    • 
        
  • WebGL graph — views/GraphView.js mounts a force-graph 2D canvas as the primary navigation surface. Subscribes to loaded/action/undo events and rebuilds {nodes, links} from the same ontology read APIs the rest of the UI uses, so security filtering, time travel, and branching all flow through it for free. Click a node to drive the detail panel via state.selection: { type, id }.
    • 
        
  • Security/Policy layer — SecurityProvider attaches to Ontology; authorizeMW is the first stage of ActionEngine's pipeline. Roles: viewer (no actions; can't see Cancelled flights), dispatcher (depart/land/delay only), admin (everything). Switch roles via the topbar; the read filter and action availability update immediately. Try security.permissions().
    • 
        
  • Apollo-sim — three environments under env/{dev,staging,prod}/manifest.json; ConfigProvider reads the ?env= URL parameter and resolves both feature flags and data-source URLs. Switching environments via the topbar dropdown reloads with a different manifest. prod locks down all mutation actions; dev opens everything.
