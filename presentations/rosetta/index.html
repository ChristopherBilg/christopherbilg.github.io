Context Decay Is a Tax on Every Multi-Repo Session


  • 11 services across as many repos
  • Each repo has its own conventions, test commands, deploy story
  • Documentation drifts the moment code changes
  • Agents either hallucinate or interrupt the engineer to re-ask
Every time you open a Claude Code session in a service repo, the agent has to figure out the same things from scratch - how do we test here, where do migrations live, what's the deploy command, what conventions apply. Documentation almost never keeps up with the code. So agents either guess wrong, or they stop and ask you. Both burn time. With 11 services, that tax adds up fast.

Cross-Service Wiring Is Where It Hurts Most


"Catch wiring drift during service-adds, not six months later."

Single-repo context decay is annoying. Cross-service context decay is dangerous. When service A changes its API contract and service B doesn't notice, you don't find out until a customer hits the bug. The pattern Rosetta exists to enforce is - catch the drift the moment a wiring change happens, not six months later.

Things We Tried First


  • CLAUDE.md / AGENTS.md in each repo - drift the moment code changes
  • Scattered READMEs - agents have to know to look at all of them
  • Pasting context into the chat by hand - burns engineer time, never consistent
  • Vector search over docs - retrieves text, misses the relationships between services
We didn't jump straight to building this. We tried the obvious things first. CLAUDE.md files are great until the code changes and nobody updates the file. Scattered READMEs assume the agent knows where to look. Manual context-pasting is a per-engineer tax, and what gets pasted varies. Vector search retrieves chunks of text but loses the structural relationship between services. None of those solved the actual problem.

What We Needed



  • Single source of truth that agents can pull from on demand
  • Updated when the code updates, not when someone remembers
  • Repo-aware, not one-size-fits-all
  • Drop-in for Claude Code - no bespoke client
So we wrote down the criteria. On-demand, current, repo-aware, and zero client work for the engineer. That's what led us to MCP. Claude Code already speaks the protocol - we just had to be the server.

Rosetta in One Slide


  • An MCP server that ships as one npm package
  • Registered once at user scope - every Claude Code session picks it up
  • Serves bootstrap payloads tailored to the repo you're in
  • Two content streams: hand-curated skills + auto-synced API contracts
Rosetta is an MCP server. Claude Code already knows how to talk to MCP servers, so there's no new tooling for the engineer. You register Rosetta once at user scope and every session automatically connects. The agent calls a single tool - bootstrap - and gets back a payload tailored to whichever repo it's working in. The content behind that payload comes from two sources: hand-written skills for workflow knowledge, and auto-synced skills for API contracts.

Three Flavors of Skills


Shared

Identical across every repo - HIPAA, auth, naming

Repo-Specific

How to test, deploy, run migrations here

Cross-Service

API contracts between this repo and the ones that call it


Same MCP server, three indexes - picked at bootstrap time based on which repo you opened.

Skills live in three categories. Shared skills are things every engineer needs - HIPAA handling, auth patterns, naming conventions. Repo-specific skills are how this particular service tests, deploys, and migrates. Cross-service skills describe the API contracts between repos. When you call bootstrap with a repo name, Rosetta picks the right slice across all three categories and ships it back.

The 5 Tools You'll See Most


Tool Purpose
bootstrap(repo) Session entry point - tailored payload
get_skill() Fetch full skill content
get_cross_repo_impact() Impact analysis when a service changes
get_map() Service registry + dependency graph
get_global_context() Org-wide standards (HIPAA, auth, naming)
You don't need to memorize these. Bootstrap is the only one most sessions actually call - the others are there when the agent decides to dig deeper. Get_cross_repo_impact and get_skill are the two I find myself watching agents reach for most often during real work.

5 More for Deeper Dives


Tool Purpose
list_skills() Search skills by repo/category
get_service_overview() Service README
search() Keyword search across all .md files
get_agent() Load specialized agent personas
get_domain() Cross-cutting domain knowledge (reserved)
These are the discovery and niche tools. List_skills and search help when the agent doesn't know what's available yet. Get_service_overview pulls the README. Get_agent loads specialized personas for cross-repo work. Get_domain is reserved for cross-cutting knowledge - the namespace exists but the content is still being authored.

Two One-Time Setup Steps

              
# 1. Authenticate with GitHub Packages
echo "//npm.pkg.github.com/:_authToken=YOUR_PAT" >> ~/.npmrc
echo "@talkiatry:registry=https://npm.pkg.github.com" >> ~/.npmrc

# 2. Register the MCP server (user scope, every session picks it up)
claude mcp add -s user rosetta -- npx -y @talkiatry/rosetta-mcp-server@latest

PAT needs read:packages - generate at GitHub → Settings → Developer settings → Personal access tokens.

Two commands, run once per machine. The first authenticates npm with GitHub Packages so npx can pull the private @talkiatry scope. The second tells Claude Code to start the MCP server on every session. Notice the dash-s user flag - that makes it global. You don't have to repeat this in every repo.

Per-Repo: The Bootstrap Reminder Hook

              
// <repo>/.claude/settings.json -- Ensure that you update <repo> to the actual repository name
{
  "hooks": {
    "UserPromptSubmit": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "echo '[rosetta] REMINDER: Call bootstrap(repo: \"<repo>\") before proceeding.'"
      }]
    }]
  }
}
This is the only per-repo step. The hook nudges the agent on every prompt - Claude Code surfaces the reminder, and the agent calls bootstrap on its own. Replace the repo placeholder with the actual repo name. That's it. Once this is in place, every session in this repo starts pre-briefed.

Cross-Repo Impact, Before You Write the Diff

  • Agent calls get_cross_repo_impact("ehr-service") before touching the API
  • Sees every downstream consumer in one shot
  • Catches breaking changes during draft, not in PR review (or later)
Picture this: you're adding a field to ehr-service. Before Rosetta, you'd write the migration, cut a PR, and find out in review that scheduling-service consumes that endpoint and expected the old shape. With Rosetta, the agent calls get_cross_repo_impact before it writes the diff - it sees every downstream consumer up-front and can plan the contract change accordingly.

API Contracts That Don't Lie


Before

Agent guesses field names from outdated README → integration breaks at runtime

After

Agent calls get_skill("cross-service", "ehr-service", "patient-api") → contract is regenerated from current source on every sync

This is the auto-synced side of Rosetta. Hand-curated skills cover the how - how to test, how to deploy. Cross-service skills cover the what - what fields exist, what types, what's required. And because rosetta-sync regenerates them from source on every PR, they don't go stale silently.

rosetta-sync: How It Stays Current


  • Update existing skills against fresh diffs
  • Discover skills the repo is missing
  • Mark stale when source no longer matches
  • Delete skills that no longer apply (cross-service only)

Sonnet first. Opus on retry. Keep the last good version if both fail.

The sync workflow is the engine that keeps the auto-synced content honest. Three phases - update what exists, discover what's missing, mark stale what no longer matches the source. We use Sonnet first because it's fast and cheap; on eval failure we retry with Opus; if both fail we keep the previous version. That last rule is important - we'd rather ship slightly stale content than regress.

What's Next



  • domains/ namespace for cross-cutting knowledge (reserved, empty)
  • ignore_discoveries to mute noisy skill suggestions in sync-config.yml
Two near-term things. The domains directory is reserved for cross-cutting knowledge that doesn't belong to any one repo - think clinical workflows that touch four services. It's empty today; the structure is in place for when we author content. Ignore_discoveries is the escape hatch for the sync workflow proposing skills the repo doesn't actually want - useful when discovery gets too noisy.
